## CRITICAL REVIEW BEFORE PROCEEDING

**Sui Move can't add fields to existing structs in an upgrade.** The
existing TUSD market (`0xe98ace0b…`) has 4 historical rides and exists
on chain with the OLD `SegmentMarketV4` layout. If the Move agent
added fields directly to the struct, the upgrade WILL break the
existing market when anything tries to read it.

Acceptable approaches the agent may have used:

1. **`sui::dynamic_field::add`** — attaches rug config to new markets
   without touching the struct layout. Existing markets get no
   rug_config dynamic field → roll_rug returns false → behavior
   identical to today.

2. **Wrapper struct** — new `SegmentMarketV4Rugged` that wraps the
   plain one. New bootstrap creates the wrapper; existing markets
   stay plain. Frontend needs to handle both types.

3. **Separate shared `RugRegistry`** keyed by market_id. Adds an
   extra shared object but cleanly separated.

4. **Bootstrap-only field with defaulting**: if Move actually allowed
   the field add (it doesn't — this would break) the agent would
   note the limitation.

**Before running step 1 (Move upgrade): READ the agent's diff on
`move/sources/segment_market_v4.move`.** Confirm one of (1), (2), or
(3) above. If they added fields directly to the struct, STOP and
work with the agent to refactor.

## Step 1 — Move upgrade

```bash
cd /Users/maxmohammadi/sui-options
# Build the upgraded package to confirm it compiles
sui move build --path move 2>&1 | tail -5

# Run all 547 + 6 new tests
sui move test --path move 2>&1 | tail -10
# expect: "Total tests: 642; passed: 642; failed: 0"

# Look at the upgrade capability id we'll use
jq '.upgrade_history[-1].upgrade_capability' deployments/testnet.json
# captured as $UPGRADE_CAP below

# Execute the upgrade — costs ~0.5 SUI
UPGRADE_CAP=0xa5bd66c01634671d92ce1ce6084074feaadc74e844f28e2f09af9ed8175cb590
sui client upgrade \
  --upgrade-capability $UPGRADE_CAP \
  --gas-budget 500000000 \
  move/ 2>&1 | tee /tmp/v4.26-upgrade.log

# Extract the new package_id from the upgrade output
NEW_PKG=$(grep -oE '"packageId": "0x[a-f0-9]{64}"' /tmp/v4.26-upgrade.log | tail -1 | grep -oE '0x[a-f0-9]{64}')
echo "new package: $NEW_PKG"

# Record in deployments.json
jq --arg pkg "$NEW_PKG" \
   --arg digest "$(grep -oE '"digest": "[A-Za-z0-9]+"' /tmp/v4.26-upgrade.log | head -1 | grep -oE '[A-Za-z0-9]{40,}')" \
   '.upgrade_history += [{
     "from_package_id": .package_id,
     "to_package_id": $pkg,
     "upgraded_at": (now | strftime("%Y%m%dT%H%M%SZ")),
     "upgrade_digest": $digest,
     "upgrade_capability": "0xa5bd66c01634671d92ce1ce6084074feaadc74e844f28e2f09af9ed8175cb590",
     "version": 4,
     "notes": "v4.26 — adds rug-pull house edge per docs/design/v2/26"
   }] | .package_id = $pkg' deployments/testnet.json > /tmp/dep.json
mv /tmp/dep.json deployments/testnet.json
```

## Rollback (if Step 3 verification fails)

```bash
# Bootstrap a new non-rugged market to replace the rugged one
RUG_CHANCE_BPS=0 bash scripts/bootstrap-tusd-market-rugged.sh
# Frontend auto-picks the latest, so this restores non-rug behavior.
# Old rugged market still on chain but unreachable from the UI.
```

The Move upgrade itself can't be rolled back, but with
`rug_chance_bps = 0` set per-market, the rug code path is a no-op
for any market that wants it disabled.
